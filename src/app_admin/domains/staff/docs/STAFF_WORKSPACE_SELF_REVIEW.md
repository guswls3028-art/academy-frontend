# 직원 관리 워크스페이스 — 상품/디자인 자체 리뷰

## 1. 잘 된 점

### 정보 구조
- **직원이 중심**이 되도록 좌(목록) · 우(헤더+탭+콘텐츠)가 명확히 분리됨.
- 선택 직원·연월이 URL로 유지되어 탭 전환 시 컨텍스트가 유지됨.
- 헤더에 이름·직위·급여유형·시급태그·월 선택·KPI가 한눈에 들어와 **“지금 누구, 몇 월, 얼마나 나왔는지”** 파악이 쉬움.

### 사용감
- 직원 클릭 → 같은 탭 유지 + `staffId` 반영 → 우측이 곧바로 해당 직원 기준으로 전환되는 흐름이 자연스러움.
- 월 이동이 헤더에만 있어서 탭을 바꿔도 동일한 “기준 월”로 통일됨.
- 좌측 리스트에 **이번달 N h · 예상 N원**이 있으면, 직원을 바꿀 때마다 대략적인 비교가 가능함.

### 디자인
- `staff-area.css`로 타이포·패널·칩·캘린더·리스트·빈 상태가 한 톤으로 정리됨.
- Inter/Pretendard, 작은 뱃지, 숫자 tabular-nums 등 **B2B 관리자 화면**에 맞는 밀도와 톤 유지.
- 과한 카드/그림자 없이 테두리·배경 비율 위주로 정리됨.

### 일관성
- 근태·비용·월마감·스냅샷·리포트가 모두 “콘텐츠만” 담당하고, 레이아웃·헤더·탭은 공통이라 구조가 단순함.

## 2. 개선 여지

### 로딩·빈 상태
- **헤더 KPI**: summary 로딩 중에 0h, 0원이 잠깐 보일 수 있음. → 로딩 시 칩에 "—" 또는 스켈레톤 처리 권장.
- **탭 콘텐츠**: 일부 탭(예: 스냅샷 없음)은 이미 안내 문구가 있음. 근태 “기록 없음” 등도 동일한 톤으로 정리되어 있음.

### 반응형
- `grid-cols-[300px_1fr]` 고정이라 **좁은 화면**에서는 좌측이 300px로 남고 우측이 좁아질 수 있음. 추후 768px 이하에서 좌 패널 접기/드로어 전환 검토 가능.

### 접근성
- 빈 상태 영역에 `role="status"` 또는 `aria-live="polite"`를 주면 스크린 리더 사용 시 “직원 미선택” 상태 전달이 더 명확해짐.
- 월 이동 버튼에 `aria-label`은 이미 있음. 탭 버튼에 `aria-selected` 반영하면 좋음.

### 성능
- **좌측 월별 요약**: 직원 수만큼 summary API 호출(useQueries). 직원이 20명 이상이면 요청이 많아질 수 있음. 추후 백엔드에 “월별 요약 목록” API가 생기면 한 번에 받아서 쓰는 편이 유리함.

### 콘텐츠 밀도
- **근태 탭**: 캘린더 + 선택일 상세 + 전체 기록 목록 + 급여 요약 카드로 정보가 많음. 첫 진입 시 “어디를 먼저 봐야 할지”를 도와주는 짧은 안내(툴팁/헬퍼)를 나중에 추가할 수 있음.
- **월 마감 탭**: 한 블록에 “상태·설명·마감 버튼”이 잘 묶여 있음. “마감 시 스냅샷 생성” 문구가 있어 의도가 드러남.

### 한국어·용어
- “예상 실지급”은 세금 3.3% 반영 후 금액이라, “예상 실지급액” 등으로 통일해 두는 편이 좋음. (이미 “예상 실지급”으로 표기되어 있어 의미는 전달됨.)

## 3. 정리

| 항목           | 평가 | 비고 |
|----------------|------|------|
| 정보 구조      | ◎   | 직원 중심, URL 유지, 헤더 일관 |
| 사용감         | ◎   | 탭 전환 시 컨텍스트 유지, 월 선택 명확 |
| 디자인 톤      | ◎   | B2B·한국 학원 SaaS에 맞는 밀도와 톤 |
| 로딩/빈 상태   | △   | 헤더 KPI 로딩 시 0 노출만 보완하면 좋음 |
| 반응형         | △   | 현재 데스크톱 우선, 좁은 화면은 추후 |
| 접근성         | ○   | 기본 라벨 있음, aria-selected 등 보강 여지 |
| 성능           | ○   | 직원 많을 때 summary N회 호출은 추후 API로 정리 가능 |

**종합**: 상품 관점에서 “직원 한 명을 골라서 근태·비용·급여를 한 화면에서 관리한다”는 목표가 잘 반영되어 있고, 디자인과 사용감은 요구한 대로 정제된 B2B 스타일로 맞춰져 있음. 헤더 로딩 처리와 접근성만 소폭 보강하면 실서비스에 쓸 수 있는 수준으로 보임.
